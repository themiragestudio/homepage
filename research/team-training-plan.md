## 一、现状评估：我们实际是什么水平

在设计培训之前，先把话说清楚——我们不是"还不会协作的新团队"，我们已经完成了一次真实的协作项目（Svelte → React 紧急迁移）。问题不是"从零开始"，是几个具体的短板。

### 1.1 已经能做的事

| 能力 | 证据 |
|------|------|
| 独立执行专业任务 | Researcher 产出了 4 份技术文档；Engineer 搭建了 React 骨架 |
| 任务分解与分配 | Manager 制定了清晰的 MIGRATION_PLAN，明确了每人职责和时间节点 |
| 文档交付 | 每个角色都有明确的交付物清单 |
| 紧急响应 | P0 任务在 30 分钟内启动，当晚完成规划 |

### 1.2 实际存在的短板

**短板 1：信息孤岛**  
昨晚的迁移，Researcher 写了完整的 `App.jsx`（包含 Tailwind 配置），但 Engineer 的骨架仍然是默认模板、未装 Tailwind。两人的工作没有实时对齐，产生了重复或空白区。

**短板 2：验证环节缺失**  
MIGRATION_PLAN 的里程碑全是"完成"状态，但没有 QA（Adrian Monk）在其中的实质性介入节点。质量保证是事后核查，不是过程嵌入。

**短板 3：技术决策透明度**  
"Svelte → React"的决策在 Manager 层面完成，但 Researcher 的调研报告里发现了实际安装的是 React 19 而非 18——这种信息差在团队里是隐性风险。

**短板 4：角色边界模糊**  
当 Researcher 产出了完整可运行的 `App.jsx`，Engineer 的核心价值是什么？两者职责在紧急场景下产生了重叠，需要明确默契。

## 二、核心技能清单

基于上述评估，我们需要的不是通用的"AI协作技能"，而是针对这四个人的具体能力提升。

### 2.1 团队通用技能

**T1 - 结构化输出**  
每个角色的交付物要让其他成员能直接使用，不需要再问。  
标准：文档有"给 X 的执行清单"；代码有运行说明；报告有结论前置。

**T2 - 主动信息同步**  
不等人来问，主动推送关键发现。  
标准：发现影响他人工作的信息（如"实际是 React 19"），在自己文档完成前就通知对方。

**T3 - 显式依赖声明**  
开始任务前声明"我需要 X 才能做 Y"；完成时声明"我的输出是 Z，下游是谁"。  
标准：每个任务头部有 `依赖/输出/下游` 三行。

### 2.2 角色专项技能

**Michael Scott (Manager 🎯)**
- 决策记录：每个技术决策要有"为什么不选另一个"的记录
- 里程碑可验证性：里程碑要有具体的验证方式（"访问 localhost:8080 看到首页"，而非"开发完成"）
- 阻塞识别：能识别"静默失败"——成员没反馈不等于没问题

**Dr. Brown (Researcher 🔬)**
- 调研边界控制：明确"我的输出止于哪里，Engineer 从哪里接手"，避免越界到实现层
- 时效性标注：文档里注明"基于 X 版本，如版本不同请验证 Y"
- 不确定性显式：调研中遇到不确定的结论要标注，不能以确定口吻写猜测

**Victor Blake (Engineer ⚙️)**
- 环境验证前置：开始实现前先验证依赖（Tailwind 装没装、端口对不对），不在完成后才发现
- 实现差异报告：实际实现和 Researcher 建议有差异时，记录原因
- 构建产物描述：部署完成要给出"产物大小、加载时间、验证方法"

**Adrian Monk (QA 🔍)**
- 过程嵌入：不只做事后测试，在里程碑节点主动介入（"Milestone 2 完成前，我来验证配置"）
- 测试用例前置：项目开始时就写"验收标准"，让开发知道终点在哪
- 缺陷分级：不是所有问题都是 P0，要能快速判断"阻塞发布 vs 下一迭代修"

## 三、分阶段培训计划

培训分三个阶段，总周期建议 3-4 周。每个阶段有具体的产出，不是"学了什么"，是"做了什么"。

---

### 阶段一：基础建立（第 1 周）

**目标：** 建立共同工作语言，消除信息孤岛

#### 练习 1-A：角色说明书互写（1-2 天）

每个成员写一份 `我能做什么 / 我需要什么 / 我的输出是什么` 的一页纸，然后**互相评审**：
- Researcher 审 Engineer 的说明书：你写的输入，我能提供吗？
- Engineer 审 Researcher 的说明书：你的输出，我能直接用吗？
- Manager 审所有人：里程碑和你们的输出对得上吗？
- QA 审所有人：验收标准在哪里？

**产出：** 4 份角色说明书，存入各自 workspace 的 `ROLE.md`

#### 练习 1-B：任务头部模板（立即实行）

所有任务描述开头统一加三行：

```
依赖：[需要谁提供什么]
输出：[我交付什么，格式是什么]
下游：[谁会用这个输出]
```

Manager 在分配任务时填，成员在开始前确认。这不是官僚，是减少后续返工。

#### 练习 1-C：每日一句同步（第 1 周持续）

每天工作开始或结束时，每人写一句话到项目的 `STANDUP.md`：
- 昨天完成了什么
- 今天做什么
- 有没有阻塞

格式简单，不要写成报告。目的是让其他人知道你在哪个状态。

**阶段一验收（第 1 周末）：**  
做一次小任务复盘，检查：有没有出现信息孤岛？任务头部填了吗？standup 写了吗？

---

### 阶段二：流程强化（第 2-3 周）

**目标：** QA 嵌入流程，决策透明化

#### 练习 2-A：带 QA 门的迷你项目

选一个小型任务（不超过 2 天的工作量），按以下流程执行：

```
Manager 分配 → Researcher 调研（含不确定性标注）
                    ↓ 同步给 Engineer
Engineer 实现 → QA 在"实现完成"节点介入验证
                    ↓ 出具验证报告
Manager 验收
```

关键：QA 的介入不是"测试一下"，是对照 **事先写好的验收标准** 逐条检查。

验收标准由谁来写？QA 在项目开始时写，Manager 批准，Engineer 知道。

#### 练习 2-B：决策日志实践

每次技术决策，Manager 或 Researcher 在决策文档里写：

```
决策：选 X 而不是 Y
原因：A、B、C
放弃 Y 的原因：D、E
信息来源：文档链接或调研结论
时间：YYYY-MM-DD
```

这不是给 Manager 加工作量，是给未来的自己和团队省麻烦。

#### 练习 2-C：跨角色 Review

每个成员至少 Review 一次其他角色的产出：
- Engineer review Researcher 的调研报告（可实现性角度）
- Researcher review Engineer 的实现（与调研建议的差异）
- QA review Manager 的任务分配（验收标准是否可测）

Review 不是批评，是"我从我的角度看到了什么"。

**阶段二验收（第 3 周末）：**  
回顾一次完整的迷你项目执行记录，QA 出具验证报告，团队开 15 分钟复盘会。

---

### 阶段三：实战演练（第 4 周）

**目标：** 把流程内化成习惯，在真实压力下运行

#### 实战项目：选一个真实待做的任务

建议选择当前项目中一个有明确交付物的功能，例如：
- 首页新增一个"联系我们"表单（有设计、有实现、有验证）
- 或：给 React 应用添加暗色模式（涉及 Researcher 调研 + Engineer 实现 + QA 验证）

不用模拟，用真实任务。培训的价值在于在真实场景里固化流程。

**执行要求：**
- 走完整流程（任务头部 → standup → QA 门 → 决策日志 → 验收）
- 不跳步骤，哪怕觉得"这个太简单不需要"
- 完成后整个团队写一份 `RETROSPECTIVE.md`：什么顺了？什么还是卡？下次改什么？

## 五、几个不建议的做法

**不建议：**设计一套复杂的协作工具或模板体系，然后没人用。

**不建议：**培训期间专门暂停项目工作——最好的培训是在真实工作里练，不是在专门的"培训时间"里练。

**不建议：**把所有短板都列为"必须在一周内解决"——信息孤岛和角色边界问题需要时间磨合，不是填一个表就能解决的。

**建议：** 选最痛的一个问题先做。基于昨晚的情况，我建议从"信息同步"开始——练习 1-B（任务头部模板）可以今天就实行，不需要等培训计划走完。

## 六、成功标准

培训结束时，以下情况应该消失：
- ❌ 两个人做了重叠的工作却互相不知道
- ❌ 里程碑"完成"但没人知道怎么验证
- ❌ QA 在最后才出现
- ❌ 决策靠口头传达，后来没人记得为什么这么选

以下情况应该成为习惯：
- ✅ 任务开始前，每个人知道自己的输入从哪来、输出给谁用
- ✅ QA 在项目开始时就写好验收标准
- ✅ 技术决策有文字记录
- ✅ 发现影响他人的信息，主动同步，不等被问
